# 模型對比測試（2026-04-03）

## 環境
- GPU: NVIDIA RTX 5070 Ti (16GB VRAM)
- CPU: AMD Ryzen 9 9950X
- RAM: 64GB DDR5
- OS: WSL2 Ubuntu
- Ollama: 0.17.4

## 測試目的
對比三種不同定位的本地模型：
- qwen3.5:9b（通用型 6.6GB）— 已有數據作為基準
- gemma3:12b（Google 多模態 8.1GB）— 新測試
- mimo-7b-rl（小米推理專精 4.7GB）— 新測試

---

## gemma3:12b（Google — 多模態 VLM）

### 基礎指標
- 模型大小：8.1GB
- 首次載入：79s
- 推理速度：85.6-86.3 tok/s
- VRAM 佔用：~12GB（估算）

### 測試結果

| Test | 結果 | Tokens | Speed | 備註 |
|------|------|--------|-------|------|
| Test 1: 基礎生成 | ✅ | 9 | 66.3 tok/s | 正確回答 2+2=4 |
| Test 2: 工具調用 | ❌ | 0 | - | 完全不回應 tool_calls |
| Test 4: 寫檔案 | ❌ | 0 | - | 無 tool_calls |
| Test 5: 讀檔+分析 | ❌ | 0 | - | 給工具就不回應 |
| Test 7: 程式碼分析 | ✅ | 935 | 85.6 tok/s | 繁中 1800字 正確分析 O(2^n) |
| Test 11: Debug 3 bugs | ✅ 3/3 | 856 | 85.8 tok/s | 繁中 1776字 全部找到 |
| 優化B: 多工具選擇 | ❌ | 0 | - | 不回應 |
| 優化D: 繁中輸出 | ✅ | 808 | 86.3 tok/s | 1334字 品質很好 |

### 評估
- 純文字分析能力：⭐⭐⭐⭐⭐（繁中品質最佳）
- 工具調用能力：⭐（標準版完全不支援）
- Agent 適用性：❌（需要 fine-tune 版 orieg/gemma3-tools:12b）
- 速度：⭐⭐⭐⭐（85 tok/s 是 qwen3.5 的 2.2 倍）

### 結論
gemma3:12b 是優秀的文字分析模型但不能當 Agent。繁中輸出品質最好，速度快，但只要給 tools 參數就完全不回應。定位：純文字顧問/分析器。

## mimo-7b-rl（小米 — 推理專精 LLM）

### 基礎指標
- 模型大小：4.7GB（最小）
- 首次載入：2.1s（幾乎瞬間 — 因為比 qwen3.5 小很多）
- 推理速度：138.6-149.1 tok/s（最快）
- VRAM 佔用：~8GB（估算）

### 測試結果

| Test | 結果 | Tokens | Speed | 備註 |
|------|------|--------|-------|------|
| Test 1: 基礎生成 | ✅ | 746 | 149.1 tok/s | 正確但 thinking 很長(英文) |
| Test 2: 工具調用 | ⚠️ | 2770 | 143.5 tok/s | 有 tool_calls 但重複呼叫了兩次 |
| Test 4: 寫檔案 | ⚠️ | 7692 | 138.6 tok/s | tool_calls 格式問題 |
| Test 5: 讀檔+分析 | ⚠️ | 4801 | 139.7 tok/s | 有回應但內容短(123字) |
| Test 7: 程式碼分析 | ✅ | 1999 | 144.9 tok/s | 簡中 3454字 分析完整 |
| Test 11: Debug 3 bugs | ✅ 3/3 | 6420 | 138.8 tok/s | 簡中 970字 全部找到 |
| 優化B: 多工具選擇 | ✅ | 454 | 147.8 tok/s | 正確選擇 web_search |
| 優化D: 繁中輸出 | ⚠️ | 667 | 148.1 tok/s | 739字 有中文但偏簡體 |

### 評估
- 純文字分析能力：⭐⭐⭐⭐（推理強 但簡中為主）
- 工具調用能力：⭐⭐⭐（有能力但不穩定 重複呼叫問題）
- Agent 適用性：⚠️（多工具選擇正確 但執行不穩定）
- 速度：⭐⭐⭐⭐⭐（149 tok/s 是 qwen3.5 的 3.8 倍）
- 大小：⭐⭐⭐⭐⭐（4.7GB 可以和工廠同時跑）

### 特殊發現
1. thinking 模式非常長 — 簡單問題也會生成 700+ tokens 的思考過程
2. 多工具場景正確選擇 web_search 而不是 bash — 判斷力有
3. 單工具場景會重複呼叫同一個命令兩次 — 需要外殼去重
4. 載入速度極快（2.1s vs qwen3.5 的 106s）

### 結論
mimo-7b-rl 是速度怪物（149 tok/s）且體積最小（4.7GB）。推理能力強，多工具選擇正確，但工具調用會重複。定位：快速推理引擎/配合外殼去重後可做輔助 Agent。

## 三模型總對比

| 指標 | qwen3.5:9b | gemma3:12b | mimo-7b-rl |
|------|-----------|-----------|-----------|
| 大小 | 6.6GB | 8.1GB | 4.7GB |
| 速度 | 39 tok/s | 85.6 tok/s | 149 tok/s |
| 首次載入 | 106s | 79s | 2.1s |
| 工具調用格式 | ✅完美 | ❌不支援 | ⚠️有但重複 |
| 多工具選擇 | ✅正確 | ❌不回應 | ✅正確 |
| 程式碼分析 | ✅ | ✅最佳繁中 | ✅最快 |
| Debug 能力 | ✅ 3/3 | ✅ 3/3 | ✅ 3/3 |
| 繁中輸出 | ✅ | ✅最佳 | ⚠️簡中為主 |
| 思考鏈 | ✅(可控) | ❌無 | ✅(過長) |
| Agent 適用性 | ⭐⭐⭐⭐⭐ | ⭐ | ⭐⭐⭐ |
| 整體推薦 | 🥇主力Agent | 🥉純文字顧問 | 🥈快速推理 |

## 最終建議

1. **小龍主力**：繼續用 qwen3.5:9b — 工具調用完美是不可替代的優勢
2. **第二模型**：mimo-7b-rl 值得留著 — 4.7GB 夠小可以和主力共存
   - 適合：快速文字分析、推理任務、不需工具調用的場景
   - 需要：外殼加工具調用去重邏輯
3. **gemma3:12b**：除非拉 fine-tune 版 否則只能當純文字助手
   - 如果要用需要 ollama pull orieg/gemma3-tools:12b

## 書籍素材價值
- 速度對比數據（39 vs 86 vs 149 tok/s）
- 工具調用能力差異（完美 vs 不行 vs 有但不穩定）
- 「最小的不一定最弱」— 4.7GB mimo 推理超過 8.1GB gemma
- 「工具調用是 Agent 的分水嶺」— gemma 分析能力最強但不能當 Agent
- 「外殼紀律」理論再次驗證 — mimo 有能力但缺自律
